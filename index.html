Prerequisite:
Website Accessibility

An accessible website is usable by as many people as possible, regardless of their ability or circumstances.

I'm sure most of you are familiar at some level with website accessibility since you've turned up for an accessibility talk. But I want to start with a simple definition of accessibility so we're on the same page.

Accessible website development is the practice of making websites usable by as many people as possible, regardless of their ability or circumstances.

Traditionally people think of this as only those with disabilities. But making websites accessible also benefits those using mobile devices, slow internet connections, or those with temporary heath issues.

Why Accessibility?

  • Legal requirements
    *I'm not a lawyer!
  • Everyone benefits from accessibility.

There are many reasons why accessibility is important but I'll briefly cover just a couple.

Legal requirements

Disclaimer, I am not a lawyer, please consult your lawyer for legal advice.

Many countries have legal accessibility requirements and penalties if you're found to not be following accessibility requirements. In the United States federal agencies and often those receiving federal grants are required to adhere to Section 508 standards. Other businesses in the United States such as local governments and those open to the public must adhere to the Americans with Disabilities Act (ADA)

I've personally been in conversations with people at WordCamps who were sued for violating accessibility requirements. It's better to do it right the first time then to get tangled up in costly litigation.

Everyone benefits from accessibility.

Accessibility benefits all users. If you develop features for users that are hard-of-hearing it will benefit users watching in loud places or with the audio turned off. If you develop features for users that are visually impaired it will benefit users the suffer from migraines and can't currently look at their screens.

As much as we'd like to avoid it, we're all aging and will most likely some day benefit from accessibility ourselves either due to age related heath issues such as visual impairment or some more extreme health issues.

Even though we may not realize it we all need accessibility.

Fully adopting accessibility first development now ensures an accessible future.

What is Accessibility First Development?

  • It's Necessary
  • It's Inclusive
  • It's Proactive

Accessibility first development is a shift in the way you approach your thinking about the development process. Choosing to create blocks, plugins, and themes that are as accessible as possible out of the box.

It's necessary

Accessibility first development embraces the idea that accessibility is a necessity. It's not added later or worse ignored altogether if the project deadline is approaching. Prioritizing accessibility takes it from being a baseline standard to a core project requirement.

It's Inclusive

Accessibility first aims to target the users with the most severe disability. This creates a trickle down affect that covers all other users with little to no disabilities.

It's Proactive

Accessibility first development addresses inclusivity at the beginning and through out the development process, not as an afterthought. Addressing accessibility earlier saves time and resources, through a more efficient development workflow.

If it's not accessible, it's broken!

Simply put, If it's not accessible, it's broken!

Developers must adopt the mindset that if what they are coding is not accessible it's broken. It should be fixed and tested before being merged into production.

The WordPress Developers Accessibility Responsibility

  • WordPress powers over 40% of all websites
  • WordPress products are content producing
  • Ensure blocks, plugins, and themes are fully accessible at launch
  • Retroactive remediation can be daunting and expensive.

WordPress developers have a unique responsibility to take a proactive approach to developing blocks, plugins, and themes that are fully accessible at launch. These products are generally content producing or duplicated on multiple websites. With WordPress' massive ~40% market share of websites, remedying accessibility issues retroactively can be a daunting in not impossible task.

For example if you create and release a block plugin that generates HTML markup in the database that is inaccessible it could be installed on any number of websites and used on any number of pages. Releasing an update to fix that issues would require all those site to update to the latest version of that plugin. Most likely an update alone wouldn't fix this issue as a find and replace would likely need to be run on the database to fix the inaccessible code.

A Word of Caution

Be cautious of layered on one click accessibility solutions.

There are some people out there that claim accessibility can be layered on top of an existing, inaccessible website (overlays). This is not true and these people are typically trying to sell you something.

There is no replacement for making accessibility part of your development knowledge base from the onset.

We've Been Here Before - Mobile First Development

  • Mobile first required a shift in thinking
  • Mobile first required adjustments in development workflows
  • Accessibility first development must become just as common place as mobile first development.

Around 2009, we first heard about the mobile first concept. It was an overwhelming shift in thinking and development workflows. Now it is just part of our daily lives as developers. I actually think it would be more work to make a website that isn't responsive at this point.

Today the accessibility first concept may seem just as daunting as mobile first did years ago. With the right tools and attitude Accessibility can become just as integral to our development workflow as mobile first.

Accessibility First Testing

Test accessibility early and test it often.

The Accessibility First Mantra: Test accessibility early and test it often.

Automated Testing

  • Reduces repetitive tasks
  • Reduces potential for human error
  • Saves time and cost

Automation is crucial to an accessibility first development workflow. It reduces repetitive tasks, and the potential for human error. Testing can quickly identify accessibility issues and provide full site reports, saving time and cost.

Accessibility Checker

(Free WordPress Plugin)
  • Accessibility error and warning notifications right on your post and/or page edit screen.
  • 40 different accessibility checks
  • No limits on pages scanned
  • Readability analysis (Flesch Kincaid grade level)
  • Ignore feature
  • All data store on your server
  • An more...

Software as a Service (Saas) Scanning Tools

  • Monsido
  • Pope Tech (WAVE API)
  • Siteimprove

Saas scanning tools such as Monsido, Pope Tech, and Siteimprove all run from an external server. These are typically premium services that will scan a whole website or from a defined sitemap. They may have time delays or limits based on different plan levels. Detail reporting and management tools are also provided.

Automated Doesn't Equal Accessible

Automation doesn't equal an accessible website. It just mean that you've made accessible anything that can be automated. Some issues must be tested manually.

  • Evaluates the rendered page
  • Allows you to evaluate web content for accessibility issues directly within your browser.

GitHub Actions

  • Axe
  • pa11y

This is a screenshot of a GitHub actions error with text that reads "All checks have failed" Followed by the failures which are axe and unit-tests.

GitHub actions can be used to automate accessibility tests with libraries like axe, pally, and lighthouse.

These actions can be run on pull requests or update as a stop-gap to ensure accessibility is tested before merging code into production. Any errors would need to resolved before being accepted.

  • axe by Deque: is a command line interface (CLI)
  • Pa11y: a command line interface (CLI) tool that you can use from your terminal
  • Lighthouse: by Google included in the Chrome browser development tools. It can be run from the terminal. It generates reports on accessibility and more.

Keyboard

Use your keyboard to move through any interactive elements. Look for:

  • Ability to tab to any interactive element (buttons or links) with the tab key.
  • Nothing invisible or hidden (that you cannot see) gets focus.
  • Focus order follows the order of the page, left to right, top to bottom.
  • Visible focus outline (2px solid with a 2px offset) that sufficiently contrasts with the background (AAA color contrast).
  • Buttons should be able to be triggered by the Enter/Return key or the space bar.
  • Links should be triggered by the Enter/Return key.
  • If a button or link triggers a scroll or opens a modal, etc. the focus should shift to the scrolled-to area or into the modal.
  • If a modal is opened, when it is closed, the focus should return to the button that triggered the modal to open.

Screen Readers

Use either NVDA (Windows) or VoiceOver (Mac) to listen to the element. Listen for:

  • Missing labels - anything that just says “button” or “link” but doesn't say what the button or link is or does.
  • Confusing or incorrect labels or alternative text. For example, a close button on a modal that just reads as “X” rather than “close modal.”
  • Ambiguous labels that don't have meaning without the surrounding text, for example, “Read More”, “Download”, etc.
  • Unnecessary clutter (multiple links in a row going to the same place, decorative icons next to linked text, etc.)
